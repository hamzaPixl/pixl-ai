# Output Schema

JSON schema for `requirements.json` and markdown structure for `project-overview.md`. Documents which downstream skills consume which files.

## requirements.json Schema

Location: `.context/spec/requirements.json`

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["source", "generated_at", "total", "by_priority", "by_category", "requirements"],
  "properties": {
    "source": {
      "type": "string",
      "description": "Path to the source PRD file",
      "example": ".context/prd.md"
    },
    "generated_at": {
      "type": "string",
      "format": "date-time",
      "description": "ISO 8601 timestamp of generation"
    },
    "total": {
      "type": "integer",
      "description": "Total number of requirements"
    },
    "by_priority": {
      "type": "object",
      "required": ["must", "should", "could"],
      "properties": {
        "must": { "type": "integer" },
        "should": { "type": "integer" },
        "could": { "type": "integer" }
      }
    },
    "by_category": {
      "type": "object",
      "required": ["functional", "non-functional", "constraint"],
      "properties": {
        "functional": { "type": "integer" },
        "non-functional": { "type": "integer" },
        "constraint": { "type": "integer" }
      }
    },
    "requirements": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["id", "category", "summary", "acceptance_criteria", "priority", "search_hints"],
        "properties": {
          "id": {
            "type": "string",
            "pattern": "^R-\\d{3}$",
            "description": "Unique requirement ID (R-001 through R-199)"
          },
          "category": {
            "type": "string",
            "enum": ["functional", "non-functional", "constraint"]
          },
          "summary": {
            "type": "string",
            "description": "One-sentence description of the requirement"
          },
          "acceptance_criteria": {
            "type": "array",
            "items": { "type": "string" },
            "minItems": 1,
            "description": "Testable criteria for verification"
          },
          "priority": {
            "type": "string",
            "enum": ["must", "should", "could"]
          },
          "search_hints": {
            "type": "array",
            "items": { "type": "string" },
            "minItems": 1,
            "description": "Code search patterns for spec-review coverage scanning"
          },
          "notes": {
            "type": "string",
            "description": "Ambiguity notes, inferred status, or clarification needed"
          }
        }
      }
    }
  }
}
```

### Requirement ID Ranges

| Range | Category | Example |
|-------|----------|---------|
| R-001 – R-099 | functional | User login, CRUD operations, notifications |
| R-100 – R-149 | non-functional | Performance targets, uptime SLAs |
| R-150 – R-199 | constraint | Tech stack mandates, compliance requirements |

## project-overview.md Structure

Location: `.context/project-overview.md`

The file follows a fixed 8-section structure with a downstream integration footer:

```
# Project Overview
> Generated by /prd-analysis from `.context/prd.md`
> Generated at: {ISO timestamp}

## 1. Project Overview          ← elevator pitch, goals, metrics, users
## 2. Domain Model              ← entities, relationships, bounded contexts
## 3. Architecture Overview     ← components, integrations, tech stack
## 4. Milestones                ← phased delivery plan
## 5. Feature Breakdown         ← epics → features with MoSCoW
## 6. Technical Requirements    ← performance, security, scalability
## 7. Risks & Assumptions       ← probability/impact matrix
## 8. Open Questions            ← blocking vs informational

---
## Downstream Integration       ← pointers to next skills
```

See `analysis-categories.md` for the detailed format of each section.

## Downstream Consumption

| File | Consumed By | How |
|---|---|---|
| `.context/prd.md` | `/task-plan` | Step 0 input parsing — reads raw PRD for context |
| `.context/prd.md` | `/prd-pipeline` | Phase 0 setup — skips PRD copy step |
| `.context/prd.md` | `/spec-review` | Mode A — source document for decomposition |
| `.context/spec/requirements.json` | `/task-plan` | Step 0 — detects pre-existing requirements, skips own decomposition |
| `.context/spec/requirements.json` | `/spec-review` | Mode C — skips decomposition, goes straight to coverage scan |
| `.context/spec/requirements.json` | `/prd-pipeline` | Phase 1.1 — skips spec decomposition step |
| `.context/project-overview.md` | `/sprint-planning` | Milestone section informs sprint goal setting |
| `.context/project-overview.md` | `/prd-pipeline` | Phase 1.4 — architecture context for implementation |

## Compatibility Notes

- The `requirements.json` format is identical to what `/spec-review` produces during its decomposition step — both use the schema from `spec-review/references/requirement-decomposition.md`
- When `/spec-review` finds a pre-existing `requirements.json`, it skips decomposition and proceeds directly to coverage scanning (Mode C)
- When `/task-plan` finds a pre-existing `requirements.json`, it uses those requirements instead of re-parsing the PRD
- The `search_hints` field is critical for `/spec-review` coverage scanning — omitting it degrades coverage detection accuracy
